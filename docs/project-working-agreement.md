## 3. 分支约定

当前所有正在进行中的任务，默认在**同一个活动分支**上推进。

规则：
- 不要为当前 in-progress 任务各自创建新分支。
- 只有在人工明确要求时，才拆分独立分支。
- 在执行前，应先确认当前所在分支，并保持与当前项目节奏一致。

## 4. 修改后的测试要求

每次代码修改完成后，必须先做与你修改范围对应的测试或验证。

原则：
- 前端改动 → 跑相关前端测试 / 页面验证
- 后端改动 → 跑相关后端测试 / API 验证
- 配置或部署改动 → 跑对应健康检查 / 验证步骤
- 文档改动 → 做链接、路径、命令可用性检查

禁止：
- 未验证直接宣称完成
- 未测试直接进入提交

## 5. Git 提交要求

只有在相关测试/验证通过后，才能执行 git 提交。

提交前要求：
- 变更范围清楚
- 测试已通过
- 没有明显未解释的脏改动

## 6. 提交后的验收要求

git 提交完成后，还必须再进行一次验收测试或验收检查。

目标：
- 确认提交后的仓库状态、结果、行为与预期一致
- 避免“测试通过但最终交付状态不对”的情况

## 7. 汇报格式要求

每次完成一轮可交付修改后，汇报至少应包含：

- 修改点
- 修改文件路径
- 测试结果
- git commit hash / message
- 验收结果
- 风险 / blocker（如有）

## 8. 使用建议

建议 Board Leader、Product Owner、各执行 agent 在当前项目推进中，把本文件视为项目级执行约定，而非平台全局默认规则。

后续如果该规则长期有效，可再决定是否抽象为更通用的协作标准。
