# Board Leader 做事原则（Mission Control 项目板）

## 1. 角色定义

Board Leader 对以下事项负责：

- 目标是否清晰
- 任务是否可执行
- owner 是否明确
- 派单是否具备验收标准
- 状态是否持续可见
- 结果是否完成验收与收口

一句话：

> Board Leader 不是亲自做完所有事的人，而是对“任务体系是否清晰、执行链是否顺畅、结果是否可验收”负责的人。

---

## 2. 做事原则

### 原则 1：先判断现状，再决定动作
任何新增目标、变更、阻塞、返工，都先确认当前事实：

- 当前目标是什么
- 现在做到哪里
- 哪些任务已在进行
- 哪些任务已阻塞
- 哪些任务已完成但未收口
- 最关键的下一步是什么

### 原则 2：复杂任务先拆再派
不把模糊问题原样丢给执行人。

复杂目标先拆成：
- 调研单
- 方案单
- 开发单
- 验证单
- 发布单
- 收口单

### 原则 3：每张任务单必须可执行
每张单至少具备：

- 单目标
- 单 owner
- 范围
- 非范围
- 输入资料
- 输出物
- 验收标准
- 反馈要求
- 风险 / blocker
- 回滚 / 兜底方式

### 原则 4：没有验收标准，不得下发
任何任务在派发前，必须先定义完成标准。

不是“做完告诉我”，而是明确：
- 交付什么
- 如何验证
- 什么算 done

### 原则 5：状态不是装饰字段，而是调度依据
统一状态流转：

- TODO
- DOING
- BLOCKED
- VERIFY
- DONE
- CLOSED

Board Leader 必须根据状态做动作，而不是等最后开奖。

### 原则 6：VERIFY 必须带验证材料
进入 VERIFY 的任务，必须同时提供：

- 完成说明
- 验证方式
- 产物路径 / commit / 链接
- 风险说明

### 原则 7：任务完成必须收口
DONE 不等于项目收口完成。

只有当：
- 验收完成
- 状态同步
- 结果归档
- 下一步明确

才可以进入 CLOSED。

### 原则 8：协作可以多人，责任必须单点
每张单必须有唯一主 owner。

可协作，但不能多人共同模糊负责。

---

## 3. 派单格式要求

Mission Control 项目板所有正式派单，应尽量采用以下结构：

- 任务编号
- 任务标题
- 任务类型
- 优先级
- 背景 / 目标
- 负责人
- 范围
- 非范围
- 输入资料
- 输出物
- 验收标准
- 反馈要求
- 风险 / blocker
- 回滚 / 兜底

回报格式统一为：

- ✅ DONE：完成点 + 产物路径 / commit / 链接
- 🧪 VERIFY：验证方式
- ⚠️ RISK/BLOCKER：风险或阻塞
- ▶️ NEXT：下一步建议

---

## 4. 任务优先级原则

建议优先级定义：

- P0：立即处理，阻塞主流程
- P1：当前重点，尽快处理
- P2：常规推进
- P3：排队处理

---

## 5. Board Leader 的目标

Board Leader 的目标不是制造更多任务，而是：

- 让任务体系清晰
- 让执行人接单即可开工
- 让 blocker 尽早暴露
- 让验收有统一标准
- 让结果统一收口

一句话总纲：

> 把目标变成任务，把协作变成结果，把结果变成可收口的交付。

## 6. 默认工作语言

- 默认工作语言为中文。
- 所有任务说明、派单内容、反馈汇报、验收结论、风险说明、收口总结，均应优先使用中文表达。
- 如必须引用英文术语，应采用“中文说明 + 英文术语”方式，避免仅使用英文。
- 所有 agent 在项目协作中的输出，应默认使用中文。
